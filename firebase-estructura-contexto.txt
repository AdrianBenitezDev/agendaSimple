6. Authentication

- El panel admin usa login con Google.
- El acceso real al panel no se resuelve solo por Auth.
- Ademas debe existir un documento `admins/{uid}` en Firestore.
- Si el documento no existe o `active != true`, el panel no deberia operar.

Flujo real de acceso:

1. La persona inicia sesion con Google.
2. El frontend obtiene su `uid`.
3. El panel consulta `admins/{uid}`.
4. Si existe y esta activo, carga datos y subscripciones.
5. Si no existe, puede intentar reclamar una invitacion desde `adminInvites` usando una Callable Function.

7. Cloud Functions v2

Archivo principal: functions/src/index.js

Exports actuales:

- healthcheck
  Endpoint HTTP simple para verificar que Functions responde.

- listPublicAdminProfiles
  Callable Function publica.
  Lee:
  - `admins` con `active == true`
  - `servicios` con `publicVisible == true`
  Devuelve perfiles publicos de administradores para el sitio publico.

- createManagedAdminAccess
  Callable Function autenticada.
  Solo la administradora principal o una cuenta habilitada puede usarla.
  Crea o actualiza una invitacion en `adminInvites/{email}`.

- listManagedAdminAccess
  Callable Function autenticada.
  Lista admins gestionados e invitaciones pendientes creadas por la admin principal.

- claimManagedAdminAccess
  Callable Function autenticada.
  Si la cuenta logueada tiene una invitacion pendiente en `adminInvites/{email}`,
  crea `admins/{uid}` y marca la invitacion como reclamada.

- submitAppointmentRequest
  Callable Function publica.
  Usa:
  - `admins/{adminId}`
  - `servicios/{serviceId}`
  - `clientes`
  - `turnos`
  Valida horario, admin activo, servicio valido y luego crea el turno.

Detalles importantes de Functions:

- region: southamerica-east1
- maxInstances: 10
- las Callable se dejan con `invoker: public` y `cors: true`
- la seguridad real se controla dentro del codigo verificando auth y permisos

9. Reglas de Firestore

Resumen funcional de firestore.rules:

- `signedIn()`
  Verifica que haya usuario autenticado.

- `adminProfileExists()`
  Exige que exista `admins/{request.auth.uid}`.

- `activeAdmin()`
  Exige que el admin exista y tenga `active == true`.

- `ownsAdmin(adminId)`
  El usuario autenticado debe coincidir con el `adminId`.

- `ownsOwnAdminProfile(adminId)`
  Permite leer o editar su propio documento de admin, pero con restricciones.

- `canManagePrincipalSections()`
  Habilita secciones principales como `salonMedia`.
  Se concede si:
  - el admin tiene `canManageAdminAccess == true`, o
  - el email del token es `37adrian38@gmail.com`, o
  - el email del token es `nataliasoledadromero27@gmail.com`

Restricciones por coleccion:

- admins
  - read: solo su propio perfil
  - create: false
  - update: solo ciertas claves editables
  - delete: false

- servicios
  - read: publico si `publicVisible == true`, o privado al admin duenio
  - create/update/delete: solo el admin duenio

- salonMedia
  - read: publico si `visible == true`, o admin principal
  - write: solo admin principal

- turnos
  - read/update/delete: solo el admin duenio
  - create: false desde cliente directo; se hace por Functions

- clientes
  - get/update/delete: depende de `adminId` y `adminIds`
  - list: restringido al admin primario del documento

- clientes/perfilesAdmin y tratamientos
  - solo el admin cuyo `uid` coincide con el doc

- stock
  - CRUD solo del admin duenio

- productos
  - read: publico
  - write: solo del admin duenio

- pagos
  - CRUD solo del admin duenio

10. Indices de Firestore

Indices compuestos declarados:

- servicios
  - publicVisible ASC, area ASC, sortOrder ASC
  - adminId ASC, sortOrder ASC

- turnos
  - adminId ASC, requestedStartAt ASC
  - adminId ASC, status ASC, requestedStartAt DESC

- clientes
  - adminIds ARRAY_CONTAINS, lastVisitAt DESC
  - adminIds ARRAY_CONTAINS, updatedAt DESC

- stock
  - adminId ASC, category ASC, updatedAt DESC
  - adminId ASC, updatedAt DESC

- productos
  - adminId ASC, updatedAt DESC

11. Firebase Storage

Reglas actuales:

- /public/{allPaths=**}
  - read: true
  - write: false

- /admins/{adminId}/{allPaths=**}
  - read y write solo si `request.auth.uid == adminId`

Paths de Storage usados por el frontend admin:

- admins/{uid}/profile/avatar.webp
- admins/{uid}/productos/{productId}/cover.webp
- admins/{uid}/salon/{salonMediaId}/slide.webp

Relacion con Firestore:

- `admins.photoUrl` y `admins.photoStoragePath`
- `productos.imageUrl` y `productos.imageStoragePath`
- `salonMedia.imageUrl` y `salonMedia.imageStoragePath`

13. Relacion entre frontend publico y Firestore

Sitio publico:

- Lee `servicios` con `publicVisible == true`
- Lee `productos`
- Lee `salonMedia`
- Consulta la Callable `listPublicAdminProfiles`
- Envia solicitudes de turnos con `submitAppointmentRequest`

Importante:

- La seccion publica de servicios depende de `servicios`, no de `admins` solamente.
- Los perfiles publicos de admins se enriquecen con la informacion de `admins`, pero la grilla de servicios nace desde los documentos de `servicios`.

14. Relacion entre panel admin y Firestore

Panel admin:

- inicia sesion con Google
- carga `admins/{uid}`
- se suscribe a colecciones filtradas por `adminId == uid`
- guarda servicios, clientes, stock y productos con el `uid` del usuario logueado

Escrituras vistas en frontend:

- servicios -> usa `adminId: state.user.uid`
- clientes -> usa `adminId` y `adminIds` del usuario logueado
- stock -> usa `adminId: state.user.uid`
- productos -> usa `adminId: state.user.uid`
- salonMedia -> usa `adminId: state.user.uid`, pero solo para admin principal

15. Estado conceptual actual

La arquitectura Firebase de este proyecto separa claramente:

- experiencia publica
- panel admin
- permisos por `uid`
- escrituras sensibles a traves de Functions o reglas estrictas

La pieza central del modelo es:

- `admins/{uid}` para habilitacion de acceso
- `servicios/{serviceId}` para el catalogo publico
- `turnos/{appointmentId}` para las solicitudes
- `clientes/{clientId}` con subperfiles por admin para personalizacion del negocio
